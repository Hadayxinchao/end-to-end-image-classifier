    CI/CD Pipeline¶

    This project uses GitHub Actions for Continuous Integration and Continuous Deployment (CI/CD).

    Overview¶

    The CI/CD pipeline consists of two main workflows:

    1. Tests & Linting (tests.yaml) - Runs on every push and PR
    2. Continuous ML (cml.yaml) - Trains models and generates reports

    Workflow 1: Tests and Linting¶

    Triggered On¶

    • Push to main or develop branches
    • Pull requests to main or develop

    What It Does¶

    graph LR
    A[Push/PR] --> B[Setup Python]
    B --> C[Install Dependencies]
    C --> D[Lint Code]
    D --> E[Run Tests]
    E --> F[Coverage Report]

    Jobs¶

    1. Code Quality Checks¶

    Flake8 - Syntax and style checking: 

    flake8 src tests --max-line-length=127

    Black - Code formatting: 

    black --check src tests

    isort - Import sorting: 

    isort --check-only src tests

    mypy - Type checking: 

    mypy src --ignore-missing-imports

    2. Unit Tests¶

    Runs all tests across multiple Python versions:

    • Python 3.8
    • Python 3.9
    • Python 3.10
    pytest tests/ -v --tb=short -m "not slow"

    3. Coverage Report¶

    Generates code coverage and uploads to Codecov:

    pytest tests/ --cov=src --cov-report=xml

    Configuration File¶

    .github/workflows/tests.yaml:

    name: CI - Tests and Linting

on:
  push:
    branches: [ main, develop ]
  pull_request:
    branches: [ main, develop ]

jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        python-version: ["3.8", "3.9", "3.10"]
    # ... (see full file)

    Workflow 2: Continuous ML (CML)¶

    Triggered On¶

    • Push to main or develop
    • Pull requests to main

    What It Does¶

    graph TD
    A[Trigger] --> B[Setup Environment]
    B --> C[Train Model]
    C --> D[Generate Metrics]
    D --> E[Create Confusion Matrix]
    E --> F[Generate Report]
    F --> G[Post PR Comment]

    Steps¶

    1. Setup Environment
    2. Install Python and dependencies

    3. Pull data with DVC (if configured)

    4. Train Model 

      python src/training/train.py \
  hyperparameters=fast \
  hyperparameters.num_epochs=3

    5. Generate Metrics

    6. Classification report
    7. Confusion matrix

    8. Training history plots

    9. Create CML Report

    10. Compiles all metrics into markdown
    11. Posts as PR comment
    12. Uploads artifacts

    Setting Up CI/CD¶

    1. Enable GitHub Actions¶

    GitHub Actions is enabled by default for public repositories.

    For private repos: 1. Go to Settings Actions General 2. Enable "Allow all actions and reusable workflows"

    2. Add Secrets (if needed)¶

    For DVC remote storage or other services:

    1. Go to Settings Secrets and variables Actions
    2. Add secrets:
    3. DVC_REMOTE_URL (if using DVC)
    4. AWS_ACCESS_KEY_ID (if using S3)
    5. etc.

    3. Configure DVC Remote (Optional)¶

    If using DVC:

    # Local setup
dvc remote add -d storage s3://mybucket/path

# In GitHub Actions, add secret
# Then modify workflow:
- name: Configure DVC
  env:
    AWS_ACCESS_KEY_ID: ${{ secrets.AWS_ACCESS_KEY_ID }}
    AWS_SECRET_ACCESS_KEY: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
  run: dvc pull

# Add more lint checks
- name: Run pylint
  run: pylint src

- name: Train model

      hyperparameters.num_epochs=10

    Best Practices¶

    1. Keep CI Fast
    2. Use fast config for CI
    3. Cache dependencies

    4. Run slow tests separately

    5. Fail Fast

    6. Run linting before tests

    7. Use continue-on-error: false for critical jobs

    8. Artifact Management

    9. Upload trained models
    10. Save reports for later review

    11. Use appropriate retention days

    12. Security

    13. Never commit secrets
    14. Use GitHub Secrets
    15. Limit workflow permissions

    Troubleshooting¶

    Tests Failing Locally But Passing in CI¶

    • Check Python version
    • Clear cache: pip cache purge
    • Check for environment-specific code

    CML Report Not Appearing¶

    • Check GitHub token permissions
    • Verify CML installation
    • Check workflow logs

    Out of Memory in CI¶

    • Reduce batch size in fast config
    • Use smaller model
    • Request more resources

    Advanced Topics¶

    Matrix Strategy¶

    Test multiple configurations:

    strategy:
  matrix:
    python-version: [3.8, 3.9, 3.10]
    os: [ubuntu-latest, windows-latest, macos-latest]

    Caching¶

    Speed up workflows with caching:

    - uses: actions/cache@v3
  with:
    path: ~/.cache/pip
    key: ${{ runner.os }}-pip-${{ hashFiles('**/requirements.txt') }}

    Scheduled Runs¶

    Run tests nightly:

    on:
  schedule:
    - cron: '0 0 * * *'  # Every day at midnight
